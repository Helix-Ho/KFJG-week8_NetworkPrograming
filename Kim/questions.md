## [ 11.4 The Sockets Interface ]

- **질문: 클라이언트와 서버가 같은 호스트 안에 있는 경우와 다른 호스트 안에 있는 경우가 무엇일까**

- 배경/맥락: 클라이언트와 서버는 물리적인 호스트 자체가 아니라, 그 안에서 실행되는 소프트웨어 프로세스
  <br>
  트랜잭션을 수행하는 두 프로세스가 같은 호스트에 있을 수도, 서로 다른 호스트에 떨어져 있을 수도 있다

- 메모(선택): 호스트 = 물리적인 컴퓨터 기계

<details>
<summary>Answer</summary>
<div markdown="1">

> - **같은 호스트에 있는 경우 = 로컬 서버 = 같은 시스템**
>   <br>내 컴퓨터에서 직접 서버 프로그램을 실행하고, 같은 컴퓨터의 웹 브라우저(클라이언트)로 해당 서버에 접속하는 상황
>   <br>`localhost`라는 도메인 이름이나 `127.0.0.1`이라는 루프백 주소를 사용하여 통신
>   <br>→ 하나의 물리적인 기계 안에서 클라이언트 프로세스와 서버 프로세스가 동시에 실행되며 상호작용

> - **다른 호스트에 있는 경우 = 원격 서버 ex) AWS … = 네트워크로 연결된 다른 시스템**
>   <br>내 컴퓨터(클라이언트 호스트)에서 웹 브라우저를 열어 AWS 등 클라우드나 데이터센터에 있는 물리적으로 떨어진 다른 컴퓨터(서버 호스트)에 접속하는 상황
>   <br>클라이언트와 서버는 서로 분리된 호스트에서 실행
>   <br>→ 컴퓨터 네트워크의 하드웨어 및 소프트웨어 자원(공인 IP, 라우터, 인터넷망 등)을 이용하여 통신

</div>
</details>

- **질문: 왜 현대 소켓 인터페이스에서 `void*` 포인터를 사용하지 않는 걸까**
- 배경/맥락: 소켓 인터페이스가 처음 개발될 당시엔 `void*` 포인터가 없어서 구조체를 넘겨줬지만, 이젠 C언어에 `void*` 포인터가 생겼는데도 여전히 구조체를 사용하는 이유
- 메모(선택):

<details>
<summary>Answer</summary>
<div markdown="1">

> 소켓 함수가 `void*` 포인터를 받게 하려면 대대적인 리팩토링이 필요
> <br>→ 현재까지 전세계적으로 **sockaddr 구조체 포인터**를 받는 형태로 수많은 애플리케이션이 개발됨
> <br>→ 소켓 함수 리팩토링 = 지금까지 개발된 모든 애플리케이션 리팩토링
> <br>⇒ 현실적으로 불가능하다

</div>
</details>

---

- **질문: 이전에 만들어진 애플리케이션들은 그대로 쓰고 이제부터 `void*` 포인터 쓰자고 정해서 새로 개발되는 앱들은 `void*` 포인터 쓰면 안 될까?**

- 배경/맥락: 이전에 구조체 전달 규약을 지켜 만들어진 애플리케이션은 그대로 두고 지금부터 `void*` 사용하기로 약속하고 개발하면 안 될까

- 메모(선택):

<details>
<summary>Answer</summary>
<div markdown="1">

> `void*` 를 도입하려면 운영체제나 C 표준 라이브러리에서 제공하는 `connect(), bind(), accept()` 같은 `기본 시스템 콜` 및 `소켓 함수의 서명(Signature)`을 변경
> <br> or 완전히 새로운 버전의 함수 세트를 만들어야 함

</div>
</details>

---

- **질문: 소켓 함수란?**

- 배경/맥락: 소켓 함수의 서명을 변경해야 해서 `void*` 도입이 어렵다는데, 소켓 함수가 정확히 뭘까

- 메모(선택): C언어 함수 서명 == 함수 원형
  <br> 컴파일러에게 함수의 존재를 알리고, 함수가 어떻게 호출되어야 하는지 정의하는 기술적인 명세

  **함수 서명의 구성 요소**

  > 반환형, 함수 이름, 매개변수 목록, 세미콜론

<details>
<summary>Answer</summary>
<div markdown="1">

> 운영체제(OS) 커널 수준에서 네트워크 통신을 위해 제공하는 가장 근본적인 `시스템 콜(System Call)`이자 `표준 인터페이스`
> <br>→ 프로그램이 운영체제의 핵심 네트워크 기능을 빌려 쓰기 위한 일종의 '창구' 역할
> <br>⇒ 모든 네트워크 애플리케이션의 가장 밑바탕엔 항상 동일한 구조의 소켓 인터페이스가 자리 잡고 있음

</div>
</details>

- **질문: 시스템 콜이란?**

- 배경/맥락: 시스템 콜을 변경해야 해서 `void*` 도입이 어렵다는데, 시스템 콜이 정확히 뭘까

- 메모(선택):

<details>
<summary>Answer</summary>
<div markdown="1">

> 사용자 프로그램이 파일 입출력, 프로세스 관리, 네트워크 통신 등 **커널 모드 권한**이 필요한 서비스를 운영체제(OS)에 요청하는 인터페이스

</div>
</details>

- **질문: 아예 새로운 OS를 만든다면 새로운 OS에선 소켓 함수에 `void*` 포인터를 넣게 구현 가능할까?**

- 배경/맥락: 이미 있는 OS들은 오랜 기간 현재의 규약을 따라왔기 때문에 바꾸기 어렵다면, 아예 새로운 OS를 만든다면 `void*`를 사용하는 쪽으로 개발할 수 있을까

- 메모(선택):

<details>
<summary>Answer</summary>
<div markdown="1">

> 이론적으로는 완전히 새로운 OS를 만든다면 소켓 시스템 콜이 `void*` 포인터를 받도록 새롭게 구현할 수 있음
> <br>하지만 이렇게 구현된 새로운 OS에선 기존 애플리케이션들이 컴파일조차 되지 않을 것
> <br>→ 완전히 새로운 OS를 개발하더라도 기존 프로그램들과의 호환성 문제로 `sockaddr` 포인터 방식을 완전히 버릴 수는 없음
> <br>→ 새로운 OS에서 기존 애플리케이션을 위해 레거시 소켓 인터페이스(`sockaddr` 포인터 방식)를 기본 제공하거나 최소한의 호환 계층(레이어)로 남겨둬야 함
> <br>⇒ 현대엔 소켓 함수를 바꾸는 대신, 애플리케이션 개발자가 `getaddrinfo` 와 같은 추상화 함수를 사용(형 변환 없이 개발 가능하도록)

</div>
</details>

- **질문: `소켓 함수(socket functions)`? `socket 함수`? 뭐가 다를까**

- 배경/맥락: CS:APP 교재를 읽다보면 `소켓 함수`도 나오고 `socket 함수`도 나오는데 문맥상 다른 개념인 것 같음

- 메모(선택):

<details>
<summary>Answer</summary>
<div markdown="1">

> - **소켓 함수(socket functions)**
>   <br> `socket, connect, bind, listen, accept` 등 네트워크 통신을 위해 사용되는 함수를 모두 통틀어 부르는 용어(인터페이스)
>   <br>socket 함수는 소켓 주소 구조체 `sockaddr`이 필요하지 않음
>   <br>소켓 함수 중 `connect, bind, accept 함수`가 인자로 `sockaddr`을 받음

> - **socket 함수**
>   <br> 소켓 함수 인터페이스에 속한 여러 함수 중 하나
>   <br> 맨 처음 `소켓 디스크립터`를 생성하는 단일 함수

</div>
</details>

- **질문: 소켓을 `끝점(endpoint)`으로 만든다는게 무슨 의미일까**

- 배경/맥락: CS:APP 교재를 읽다보면 소켓을 `끝점(endpoint)`으로 만든다는 내용이 나오는데 이게 무슨 의미인지 모르겠음

- 메모(선택): 끝점 = 종단점 = endpoint

<details>
<summary>Answer</summary>
<div markdown="1">

네트워크 통신에서 **데이터를 주고받는 양쪽 끝의 최종 출입구**를 생성한다는 의미

- **고유한 식별 주소 부여( ≠ 소켓 디스크립터)**

  > 컴퓨터의 위치를 나타내는 **IP 주소**
  > <br>특정 프로세스를 식별하는 **포트 번호**

      둘을 결합한 네트워크상의 실제 목적지 주소로 고유한 소켓 주소를 부여

- **연결의 한 쌍(Pair)**

  > 클라이언트 측 소켓(끝점 1)
  > <br>서버 측 소켓(끝점 2)

      둘이 한 쌍(Socket pair)을 이루어 양방향으로 통신

- **파일로서의 입출력**
  > 네트워크 통신 끝점 소켓 == 파일 디스크립터를 가진 하나의 열린 파일
  > <br>연결이 수립된 후에는 끝점을 통해 데이터를 읽고 씀

</div>
</details>

---

- **질문: 소켓 디스크립터(Socket Descriptor)란?**

- 배경/맥락: 네트워크 endpoint에 부여되는 고유 식별 주소가 소켓 디스크립터인줄 알았는데 아니었음

- 메모(선택):

<details>
<summary>Answer</summary>
<div markdown="1">

운영체제가 네트워크 소켓을 제어하기 위해 프로그램에 부여하는 **고유한 식별 번호(양의 정수 값)**

> <br>→ socket 함수가 성공적으로 실행되었을 때 반환하는 양의 정수값
> <br>→ 이 값을 통해 컴퓨터 내부의 텍스트 파일을 읽고 쓰듯이 네트워크 데이터를 주고 받음
> <br>⇒ 외부 네트워크의 다른 컴퓨터가 끝점을 찾아 올 땐 고유한 소켓 주소 사용

> > <br>클라이언트가 서버를 찾을 때 or 서버가 클라이언트에게 응답을 보낼 때 모두 이 고유 소켓 주소를 사용함
> > <br>네트워크 연결: 클라 끝점과 서버 끝점 한 쌍의 통신 = 양쪽 모두 고유한 식별 주소 있음
> > <br>→ 클라가 서버에게 연결 요청할 때 = 목적지 서버의 소켓 주소 알아야 함
> > <br>→ 서버가 클라에게 응답할 때 = 자신에게 연결 요청한 클라의 소켓 주소 알아야 함
> > <br>⇒ 내 프로그램 안에서 네트워크 입출력을 지시할 땐 소켓 디스크립터 번호를 사용

</div>
</details>
